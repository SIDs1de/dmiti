# Начало работы

1. Клонируй репозиторий
```bash
git clone <url>
cd dmiti_project
```
2. Установи Poetry (один раз на компьютер)
```bash
pip install poetry
```

3. Установи зависимости проекта
```bash
poetry install
```

4. Активируй виртуальное окружение
```bash
poetry env activate # получаешь команду которая начинается с "source"
# вставляешь команду и попадаешь в оболочку
# все действия необходимо выполнять в этой оболочке
# именно в этой оболочке установлены зависимости для python проекта
```

# Работа
**НИКТО не изменяет main ветку, не пушит туда**

**Каждый работает в своей ветке и по окончании работы пушит, а затем делает ПР**

ПРы проверяю лично я (Игорь), оставляю комментарии. Вы вносите
исправления, затем я проверяю заново. Когда и только когда
я ставлю approve (одобрение) на ПР - вы лично его мёрджите

Если случаются мёрдж конфликты - пишите тому, кто может
помочь в их решении. При нашей архитектуре их не должно происходить,
а значит, что это будет нестандартный случай, либо ваш косяк,
который нужно будет аккуратно порешать

### Шпаргалка
```bash
# Создать ветку и перейти в неё:
git checkout -b название

# Перейти в другую ветку 
git checkout название
# правила названия ветки ниже

# Добавить файлы в индексацию
git add . # точка - добавить все файлы.
# если хочется проиндексировать только часть, можно вместо точки писать пути

# Сделать коммит
git commit -m "сообщение" 
# сообщения обязательно должны быть осмысленными
# пример: git commit -m "feat: добавлен метод для сложения двух натуральных чисел"
# feat - фича (новый функционал), fix - фикс (исправление), других неймингов не будет
# веткам так же давать осмысленные названия. Для примера выше это может быть:
# git checkout -b "feat/addition-of-natural-numbers"

# Стянуть чужие изменения
# ДЕЛАТЬ ЭТО КАЖДЫЙ РАЗ КОГДА ВЫ САДИТЕСЬ ДЕЛАТЬ НОВЫй ФУНКЦИОНАЛ
git pull origin main
# то есть эта команда исполняется в двух случаях
# 1. Я закончил делать фичу, закоммитил. Затем делаю pull, проверяю, что не возникло коммитов, а затем пушу
# 2. Я много часов не сидел за проектом, начинаю новую фичу (НЕ В ПРОЦЕССЕ РАЗРАБОТКИ)
# Для пункта 1 команда исполняется в той ветке, в которой вы работаете
# Для пункта 2 команда исполняется в ветке main

# Пуш после коммита
git push origin название_ветки

# Проверить ваш статус (важно использовать в процессе работы, 
# чтобы понимать, что у вас закоммичено, что в индексации, а что не тронуто)
git status
```

### Как работать с ветками
У нас главная ветка main. Вы от неё создаёте свою ветку с фичей или фиксом.
Там вы работаете и заливаете по окончании в репозиторий.
Затем создаёте ПР (в main ветку). Чтобы делать новую фичу (а у вас будет много
фич, потому что 1 фича - 1 метод например), вы возвращаетесь в main
ветку, пуллите и создаёте из неё снова ветку.

# Устройство проекта
src/ - основная папка для работы  
tests/ - папка для тестов  
pyproject.toml - для контроля пакетов  
Остальные файлы на внешнем уровне являются служебными

# Запуск тестов
Тесты должны называться со слова test
```bash
# Пример
poetry run pytest tests/natural/ -v # запускается конкретный тест
# Пример 2
pytest # запускаются сразу все тесты
```

# Разработка
*demo.py* - общий файл. Там можно тестировать ваши модули. Смело
удаляйте чужой код и вставляйте свой. Не имеет значения,
чей код будет лежать в гитхабе там. Но лучше удалять оттуда
всё перед сдачей работы

Для запуска demo.py активируйте оболочку (команды выше), затем запустите
файл через ваш IDE или выполните команду
```bash
python demo.py
# или
python3 demo.py
```

# Другие правила
1. Разработка в стиле ООП
2. Обязательно прописывать типы принимаемых значений и возвращаемых
3. При использовании нейросетей внимательно читать весь код, дорабатывать его, удалять бессмысленные комментарии
